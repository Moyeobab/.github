## 1. 개요

운영 중 발생하는 장애를 실시간으로 감지하고 담당자에게 알림을 전달하는 시스템을 구축합니다.

### 1.1 현재 상황

- CD 파이프라인에서 배포 시점 알림만 존재 (배포 성공 여부, 배포 실패 시 자동 롤백 성공 여부)
- 운영 중 발생하는 장애 감지 및 알림 시스템 없음

### 1.2 목표

- 운영 중 장애 발생 시 실시간 알림
- 장애 감지 기준에 따른 알림 규칙 적용
- Discord 채널로 알림 전달

---

## 2. 알림 대상

장애 감지 기준 정의서에서 정의한 장애 중 자동 알림 대상은 다음과 같습니다.

| 장애 상황 | 감지 방법 | 자동 알림 |
|----------|----------|:--------:|
| 헬스체크 실패 | UptimeRobot (외부) | 자동 알림 |
| 핵심 API 500 에러 | Cron + 로그 스캔 (내부) | 자동 알림 |
| 그 외 장애 | 로그 확인 / 사용자 신고 | 수동 확인 |

---

## 3. 도구 선정

### 3.1 검토한 옵션

| 방식 | 헬스체크 감지 | API 500 에러 감지 | 인프라 종속성 | 비용 |
|-----|:------------:|:----------------:|:------------:|:----:|
| CloudWatch + SNS | Route 53 필요 | Logs Metric Filter 필요 | AWS 종속 | 유료 |
| Prometheus 스택 | Blackbox Exporter | PromQL 연산 | 별도 서버 필요 | EC2 추가 비용 |
| UptimeRobot + Cron | 외부 HTTP 폴링 | 로그 파싱 | 인프라 독립적 | 무료 |

### 3.2 선택: UptimeRobot + Cron

**선택 이유:**

- MAU 500미만 규모에서 별도 인프라 구축 없이 실용적
- 외부 상태 확인(Outside-in) + 내부 정기 검사(Inside-out) 조합
- 무료 사용 가능
- Discord 웹훅 연동 지원

**역할 분담:**

| 도구 | 역할 | 감지 대상 |
|-----|------|----------|
| UptimeRobot | 외부에서 서비스 가용성 감시 | 헬스체크 실패, EC2 다운 |
| Cron + 스크립트 | 내부에서 로그 기반 에러 감지 | API 500번대 에러 |

## 4. 시스템 아키텍처

```
[외부 감시 - UptimeRobot]
UptimeRobot (5분 주기)
    ↓ (헬스체크 실패 감지)
Discord 웹훅
    ↓
Discord #alerts 채널

[내부 감시 - Cron]
EC2 Cron (1분 주기)
    ↓ (로그 스캔)
check_error.sh 스크립트
    ↓ (500번대 에러 임계치 초과 시)
Discord 웹훅
    ↓
Discord #alerts 채널
```

## 5. 구축 내용

### 5.1 UptimeRobot 설정 (외부 가용성 감지)

#### 5.1.1 모니터 생성

| 항목 | 값 |
|-----|---|
| Monitor Type | Keyword |
| Friendly Name | moyeobab-api-health |
| URL | https://api.moyeobab.com/actuator/health |
| Keyword | UP |
| Keyword Type | Keyword Exists |
| Monitoring Interval | 5분 (무료 플랜) |

#### 5.1.2 Alert Contact 설정

| 항목 | 값 |
|-----|---|
| Alert Contact Type | Webhook |
| Webhook URL | Discord 웹훅 URL |
| POST Value | `{"content": "[DOWN] moyeobab-api 헬스체크 실패! 즉시 확인이 필요합니다."}` |


### 5.2 Cron + 스크립트 설정 (내부 에러 감지)

#### 5.2.1 에러 체크 스크립트

**파일 위치:** `/home/ubuntu/scripts/check_error.sh`

```bash
#!/bin/bash

# 설정
SERVICE_NAME="moyeobab-api"
ERROR_THRESHOLD=10
DISCORD_WEBHOOK_URL="YOUR_DISCORD_WEBHOOK_URL"

# 최근 1분간 로그에서 500번대 에러 카운트
ERROR_COUNT=$(journalctl -u $SERVICE_NAME --since "1 minute ago" 2>/dev/null | grep -c "500")

if [ "$ERROR_COUNT" -gt "$ERROR_THRESHOLD" ]; then
    curl -s -X POST -H "Content-Type: application/json" \
        -d "{\"content\": \"⚠️ [$SERVICE_NAME] 최근 1분간 500번대 에러 ${ERROR_COUNT}건 발생! 롤백 검토가 필요합니다.\"}" \
        "$DISCORD_WEBHOOK_URL"
fi
```

#### 5.2.2 실행 권한 부여

```bash
chmod +x /home/ubuntu/scripts/check_error.sh
```

#### 5.2.3 Cron 등록

```bash
crontab -e
```

추가할 내용:
```
* * * * * /home/ubuntu/scripts/check_error.sh
```

## 6. 알림 메시지 형식

### 6.1 헬스체크 실패 (UptimeRobot)

```
[DOWN] moyeobab-api 헬스체크 실패! 즉시 확인이 필요합니다.
```

### 6.2 서비스 복구 (UptimeRobot)

```
[UP] moyeobab-api 서비스가 정상 복구되었습니다.
```

### 6.3 API 500 에러 증가 (Cron)

```
[moyeobab-api] 최근 1분간 500번대 에러 10건 발생! 롤백 검토가 필요합니다.
```

## 7. 기존 CD 알림과의 관계

| 알림 종류 | 담당 시스템 | 시점 |
|----------|------------|------|
| 배포 성공/실패 | CD 파이프라인 (GitHub Actions) | 배포 시 |
| Smoke Test 실패 → 자동 롤백 | CD 파이프라인 (GitHub Actions) | 배포 시 |
| 헬스체크 실패 | UptimeRobot | 운영 중 |
| API 500번대 에러 증가 | Cron 스크립트 | 운영 중 |

## 8. 한계 및 대응 방안

| 한계 | 대응 |
|-----|------|
| UptimeRobot 무료 플랜은 5분 간격 | MAU 500 규모에서는 충분 |
| Cron은 EC2 다운 시 동작 안 함 | UptimeRobot이 외부에서 감지 |
| recommend 서비스 상세 모니터링 불가 | 헬스체크 엔드포인트 추가 시 UptimeRobot 모니터 추가 |
