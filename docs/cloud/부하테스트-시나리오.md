## 1. 개요

이 문서는 Moyeobab 서비스의 점심/저녁 시간대 트래픽 스파이크 상황에서 병목 지점을 식별하기 위한 부하 테스트 시나리오를 정의한다.

**핵심 목표**: "점심/저녁 시간대에 트래픽이 몰릴 때, 어디서 터지는가?"를 찾는 것이다.

---

## 2. 왜 스파이크 부하 테스트가 필요한가

### MAU 500 미만 서비스에서의 부하 테스트 의미

MAU 500 미만의 소규모 서비스라면 "부하 테스트가 필요한가?"라는 의문이 들 수 있다. 평균 트래픽만 보면 서버 한 대로도 충분히 감당 가능한 수준이기 때문이다.

하지만 Moyeobab은 일반적인 서비스와 트래픽 패턴이 다르다.

### Moyeobab의 트래픽 특성

**시간 집중형 서비스**: 하루 24시간 중 실제 사용이 발생하는 시간은 점심(11:30\~13:30)과 저녁(17:30\~19:30)으로 예상된다.

**그룹 동시 접속**: 한 모임당 5~10명이 거의 동시에 접속한다. 호스트가 카톡에 초대 코드를 공유하면 "밥 뭐 먹지?" 하면서 단체로 서비스에 접속할 수 있다.

**마감 압박으로 인한 동시성**: 투표 마감 시간이 정해져 있어서, 마감 직전에 투표 제출이 동시에 몰릴 수 있다.

### 소규모 서비스에서 장애가 더 치명적인 이유

- 사용자 한 명 한 명의 경험이 전체 평판에 직결된다
- "점심시간에 앱이 안 돼서 그냥 아무데나 갔어"라는 경험 한 번이면 재사용률이 급락한다
- 특히 그룹 서비스는 한 명이 이탈하면 그 그룹 전체가 이탈할 가능성이 높다

### 테스트 목적 요약

따라서 이 부하 테스트의 목적은 "대규모 트래픽 감당"이 아니라, **피크 시간대에 터지지 않는 최소한의 안정성 확보**이다. 구체적으로는 다음을 확인한다.

1. 점심/저녁 스파이크 시 어느 API에서 병목이 발생하는가
2. 현재 인프라로 동시 접속 몇 명까지 버틸 수 있는가
3. 병목 지점의 원인이 DB인가, 애플리케이션 로직인가, 인프라 설정인가

## 3. 테스트 대상 구간

Moyeobab 서비스 흐름에서 스파이크 시 병목이 발생할 가능성이 높은 구간 3가지를 선정했다.

### 서비스 흐름과 테스트 구간 매핑

```
점심 11:30 ─────────────────────────────────────────→ 13:00

[11:30~11:45]          [11:45~12:15]          [12:15~12:30]
호스트가 모임 생성         스와이프 투표 진행             결과 확인
초대 코드 공유            마감 직전 제출 몰림            최종 선택
참여자 동시 입장
      ↓                      ↓                      ↓
    구간 1                  구간 2                  구간 3
```

---

### 구간 1: 모임 참여 집중

**상황**: 호스트가 카톡에 초대 코드를 뿌리면, 그룹원 5~10명이 거의 동시에 참여 버튼을 누름

**테스트 대상 API**:

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | /api/v1/participate_meetings | 초대 코드로 모임 참여 |
| GET | /api/v1/meetings/{meetingId} | 참여 후 모임 상세 조회 |

**왜 병목이 예상되는가**

같은 모임에 동시 참여 시 participantCount 증가 로직에서 DB 락 경합이 발생할 수 있다. 특히 점심시간에 수십 개 모임에서 이게 동시에 일어나면 커넥션 풀이 부족해질 수 있다.

**확인할 병목 포인트**:
- DB 락 대기 시간
- 커넥션 풀 사용률
- 참여자 수 정합성 (동시 참여 시 중복/누락 여부)

---

### 구간 2: 투표 제출 동시 집중

**상황**: 투표 마감 5분 전, 아직 투표 안 한 사람들이 "아 맞다 투표" 하면서 동시에 제출

**테스트 대상 API**:

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/candidates | 투표 후보 목록 조회 |
| POST | /api/v1/meetings/{meetingId}/votes/{voteId}/submissions | 투표 제출 |
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/status | 투표 현황 폴링 |

**왜 병목이 예상되는가**

투표 제출은 쓰기 작업이라 읽기보다 무겁다. 같은 투표에 여러 명이 동시 제출하면 중복 체크 + 집계 로직에서 동시성 이슈가 생길 수 있다. 그리고 status 폴링이 계속 들어오면서 읽기 부하까지 겹친다.

**확인할 병목 포인트**:
- 쓰기 작업 응답시간
- 중복 투표 발생 여부
- 폴링 요청과 제출 요청의 리소스 경합

---

### 구간 3: 결과 조회 폭주

**상황**: 투표가 마감되고 결과가 나오면, 모든 참여자가 "뭐 나왔어?" 하면서 동시에 결과 화면 진입

**테스트 대상 API**:

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/status | COUNTED 상태 확인 |
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/results | Top3 결과 조회 |
| GET | /api/v1/meetings/{meetingId}/final-selection | 최종 선택 조회 |

**왜 병목이 예상되는가**

결과 조회는 집계 쿼리가 포함될 가능성이 높다. 투표 데이터를 그룹핑하고 정렬하는 연산이 동시에 수백 번 요청되면 DB CPU가 튈 수 있다. 캐싱이 안 되어 있으면 특히 취약하다.

**확인할 병목 포인트**:
- 집계 쿼리 응답시간
- DB CPU 사용률
- 캐싱 적용 여부에 따른 차이

## 4. 테스트 시나리오

### 부하 패턴 설계 원칙

모든 구간에서 동일한 패턴을 적용한다: **급증 → 피크 유지 → 감소**

이는 실제 점심시간 트래픽 패턴을 반영한 것이다. 11시 30분쯤 갑자기 트래픽이 치솟고, 12시~12시 30분까지 높은 상태를 유지하다가, 점심 식사가 끝나면서 서서히 줄어든다.

 20 ┼─────╯                       ╰─────

## 5. 테스트 실행 계획

### 1단계: 구간별 독립 테스트

각 구간을 개별적으로 테스트하여 구간별 한계점을 파악한다.

| 순서 | 테스트 | 예상 소요 시간 |
|------|--------|----------------|
| 1-1 | 시나리오 1 (모임 참여) | 10분 |
| 1-2 | 시나리오 2 (투표 제출) | 12분 |
| 1-3 | 시나리오 3 (결과 조회) | 8분 |

**1단계 완료 조건**: 각 구간에서 병목이 발생하는 VU 수 또는 "현재 설정으로는 병목 없음" 확인

### 2단계: 복합 시나리오 테스트

실제 점심시간처럼 여러 구간이 겹치는 상황을 테스트한다.

| 순서 | 테스트 | 상황 | 예상 소요 시간 |
|------|--------|------|----------------|
| 2-1 | 시나리오 1 + 2 | 모임 참여와 투표가 겹침 (점심 초반) | 15분 |
| 2-2 | 시나리오 2 + 3 | 투표 제출과 결과 조회가 겹침 (점심 후반) | 12분 |

**2단계 완료 조건**: API 간 리소스 경합으로 인한 추가 병목 여부 확인

## 6. 병목 판단 기준

### 응답시간 기준

| 상태 | P95 응답시간 | 의미 |
|------|--------------|------|
| 정상 | < 200ms | 터지지 않음 |
| 주의 | 200ms ~ 500ms | 병목 징후, 모니터링 필요 |
| 경고 | 500ms ~ 1초 | 병목 발생, 원인 분석 필요 |
| 위험 | > 1초 | 터짐, 즉시 개선 필요 |

### 에러율 기준

| 상태 | 에러율 | 의미 |
|------|--------|------|
| 정상 | < 0.1% | 터지지 않음 |
| 주의 | 0.1% ~ 1% | 간헐적 실패 발생 |
| 경고 | 1% ~ 5% | 사용자 체감 시작 |
| 위험 | > 5% | 터짐 |

### 병목 원인 식별 가이드

테스트 결과 "터졌다"면, 다음 증상으로 원인을 구분한다.

**DB가 원인인 경우**:
- 응답시간이 VU 증가에 비례해서 선형으로 증가
- 커넥션 풀 사용률 90% 이상
- slow query 로그 급증

**애플리케이션이 원인인 경우**:
- CPU 사용률 80% 이상에서 응답시간 급증
- GC 빈도 증가
- 특정 API만 느려지고 다른 API는 정상

**인프라가 원인인 경우**:
- 모든 API가 동시에 느려짐
- ALB 대기열 증가
- 504 Gateway Timeout 비중 높음

## 부록 A: VU(Virtual User)란

VU는 동시에 서비스를 사용하는 가상의 사용자 수다. 실제 사람이 앱에서 요청을 보내는 것처럼 행동하는 시뮬레이션 단위라고 보면 된다.

100 VU = 100명이 동시에 앱을 열고 모임 조회, 투표 제출 등의 행동을 수행하는 상황

### 왜 현재 규모보다 높은 VU로 테스트하는가

이 테스트의 목적은 **\"현재 트래픽을 버티는가\"가 아니라 \"어디까지 버티는가\"를 찾는 것**이다.

현재 MAU 500 규모에서 발생하는 실제 부하보다 훨씬 높은 VU로 테스트하면 다음을 알 수 있다:

1. **현재 인프라의 한계점**: 몇 VU에서 병목이 발생하는지
2. **성장 여유분**: MAU가 몇 배 늘어나도 괜찮은지
3. **병목 지점 우선순위**: 어느 API가 먼저 터지는지 → 개선 순서 결정

한계점을 모르면 서비스가 성장했을 때 갑자기 터지는 상황을 맞는다. 미리 한계를 알아두면 그 전에 대응할 수 있다.

## 부록 B: 왜 P95인가

### "터지는 것"과 "느린 것"은 다르다

부하 테스트에서 보는 지표는 크게 두 가지다:

| 지표 | 의미 | 사용자 체감 |
|------|------|-------------|
| 에러율 | 요청이 실패한 비율 | "앱이 안 돼" → 이탈 |
| 응답시간 (P95) | 95%의 요청이 이 시간 안에 완료 | "좀 느리네" → 기다림 |

앞서 "소규모 서비스에서 장애가 치명적"이라고 한 건 **에러율** 이야기다. 앱이 아예 안 되면 "그냥 아무데나 가자"가 되고, 그룹 전체가 이탈한다.

반면 P95는 **응답시간** 지표다. 95%가 빠르게 응답받으면, 나머지 5%는 조금 느리더라도 기다리면 된다. 느린 건 참을 수 있지만, 안 되는 건 참을 수 없다.

### P95의 의미

"100명 중 95명은 이 시간 안에 응답받았다"

나머지 5%는 네트워크 지연, 디바이스 문제 등 서버 외적 요인이 섞여 있을 수 있어서, P95가 서버 성능을 더 순수하게 반영한다.

### P99는 언제 쓰는가

P99는 **1%의 지연도 허용 안 되는 서비스**에서 사용한다.

| 서비스 유형 | P99를 쓰는 이유 |
|-------------|-----------------|
| 결제/금융 | 결제 중 10초 지연 = 이탈 + 클레임 |
| 실시간 게임, 주식 거래 | 1%의 지연도 승패/손익에 영향 |
| SLA 계약이 있는 B2B | "99% 요청은 500ms 이내" 같은 계약 조건 |

### Moyeobab에서 P95가 적합한 이유

- **응답시간 P95**: 대부분이 빠르면 그룹 경험 유지됨. 5%가 조금 느려도 기다리면 해결.
- **에러율은 별도로 1% 미만 목표**: "느린 것"은 괜찮지만 "터지는 것"은 안 됨. 에러율은 엄격하게 관리.
- **비용 효율**: P99까지 맞추려면 인프라 오버 프로비저닝 필요. MAU 500 규모에서 그 비용을 감당할 이유 없음.
