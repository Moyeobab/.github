## 1) 배경

현재 모여밥 서비스는 단일 EC2 인스턴스(2 vCPU, 2 GiB)에 Nginx + Spring Boot + PostgreSQL + Redis가 모두 배포되어 있다. Multi-Instance HA 아키텍처 전환 중 데이터베이스 마이그레이션은 전체 전환에서 Risk Level이 가장 높은 단계이다.

스파이크 트래픽 상황을 전제로 마이그레이션을 진행하므로, 수 초의 다운타임도 허용하지 않는 **완전 무중단 전환**을 목표로 한다. 롤백 역시 무중단이어야 한다.

### 저장 데이터 현황

| 구분 | 저장소 | 데이터 | 특성 |
|------|--------|--------|------|
| 서비스 데이터 | PostgreSQL | 모임, 투표, 채팅, 정산 등 | 영속 데이터, 유실 불가 |
| 세션 | Redis | 리프레시 토큰 (로그인 세션) | TTL 있음, 유실 시 재로그인 |
| 임시 데이터 | Redis | 회원가입 개인정보 동의 처리 | 짧은 TTL, 유실 시 재시도 |

## 2) 마이그레이션 방식 비교

PostgreSQL → PostgreSQL 동일 엔진 마이그레이션에서 고려한 3가지 방식:

### 2.1. Streaming Replication (물리적 복제)

```
Old DB (Primary) ──WAL 바이트 전송──→ New DB (Standby, Read-Only)
                                        │
                                   pg_promote
                                        │
                                   New DB (Primary)
```

WAL을 바이트 단위로 그대로 전송한다. New DB는 Old DB의 물리적 거울이며, promote 전까지 **Read-Only**이다.

- 시퀀스: WAL에 포함되어 **자동 복제**됨. promote 후 별도 작업 불필요.
- 전환: `pg_promote` 실행 (수 초 다운타임 발생)
- 롤백: **구조적으로 어려움**. Old DB를 다시 Standby로 만들려면 `pg_basebackup`부터 재구성 필요.

### 2.2. Logical Replication (논리적 복제)

```
Old DB ──변경사항 (INSERT/UPDATE/DELETE)──→ New DB (Read-Write 가능)
           테이블 단위, SQL 레벨 재생
```

변경사항을 SQL 레벨로 디코딩하여 전송한다. New DB는 Replication을 받으면서도 **독립적으로 쓰기 가능**하다.

- 시퀀스: **복제되지 않음**. 수동 동기화 필요.
- 전환: PgBouncer PAUSE/RESUME으로 **무중단 전환** 가능.
- 롤백: 역방향 Subscription으로 **무중단 롤백** 가능.

### 2.3. Dual Write (이중 쓰기)

```
Application ──쓰기──→ Old DB
    │
    └──────쓰기──→ New DB
```

애플리케이션 코드에서 두 DB에 동시에 쓴다. DB 레벨이 아닌 **애플리케이션 레벨** 전략이다.

정합성 리스크가 높다:
- **부분 실패**: Old 성공 / New 실패 → 데이터 불일치
- **순서 역전**: 동시 요청에서 양쪽 DB의 쓰기 순서가 다름
- **트랜잭션 불일치**: Old 커밋 / New 롤백 → 복구 불가능

이를 해결하려면 분산 트랜잭션(2PC)이나 보상 트랜잭션이 필요하며, 모든 Repository/DAO 레이어에 이중 쓰기 로직을 삽입해야 한다. 마이그레이션 완료 후 전부 제거해야 하는 일회용 코드.

### 2.4. 비교 요약

| | Streaming | Logical | Dual Write |
|---|---|---|---|
| 복제 레벨 | WAL 바이트 (물리적) | SQL 변경사항 (논리적) | 애플리케이션 코드 |
| 시퀀스 복제 | 자동 | 수동 | 앱이 양쪽에 쓰므로 자동 |
| 전환 다운타임 | 수 초 (pg_promote) | **무중단** (PgBouncer) | 무중단 (읽기 전환) |
| 데이터 정합성 | 바이트 동일 보장 | 논리적 동일 (검증 필요) | **보장 안 됨** |
| 무중단 롤백 | **불가** | **가능** | 코드 롤백 필요 |
| New DB 사전 설정 | 불가 (Read-Only) | 가능 (시퀀스 등) | 해당 없음 |
| 설정 복잡도 | 낮음 | 중간 | 높음 (코드 수정) |
| 추가 도구 | 없음 | PgBouncer | 없음 |
| 버전 제약 | 동일 메이저 버전 | PostgreSQL 10+ | 없음 |

### 2.5. 선택: Logical Replication + PgBouncer

**선택 근거:**

1. **무중단 전환**: PgBouncer PAUSE/RESUME으로 앱 재시작 없이 DB 연결 대상을 전환할 수 있다. Streaming의 pg_promote는 수 초의 다운타임이 발생하며, 스파이크 트래픽 환경에서 허용 불가.
2. **무중단 롤백**: 양방향 Logical Replication(Forward/Reverse Subscription)을 구성하면 롤백도 PgBouncer PAUSE/RESUME으로 무중단 수행 가능. Streaming은 롤백 시 pg_basebackup 재구성이 필요하여 무중단 롤백이 구조적으로 불가.
3. **사전 준비 가능**: New DB가 Read-Write이므로 Replication 수신 중에도 시퀀스 설정, 인덱스 생성 등 사전 작업 가능.

**CDC 도구(Debezium 등)는 사용하지 않는다.** PostgreSQL → PostgreSQL 동일 엔진이므로 PostgreSQL 내장 Logical Replication(Publication/Subscription)으로 충분하다. Debezium은 Kafka가 필요하여 현재 인프라에서 오버엔지니어링이다.

## 3) PostgreSQL 마이그레이션 상세

### 3.1. 아키텍처

```
[현재 — AS-IS]
  App (HikariCP) → Old PostgreSQL (단일 인스턴스 내)

[마이그레이션 중]
  App (HikariCP) → PgBouncer → Old PostgreSQL
                                  │ Forward Logical Replication
                                  ↓
                               New PostgreSQL (별도 인스턴스)

[전환 후]
  App (HikariCP) → PgBouncer → New PostgreSQL
                                  │ Reverse Logical Replication (롤백 대비)
                                  ↓
                               Old PostgreSQL
```

### 3.2. 사전 준비

#### PgBouncer 도입

PgBouncer를 App과 DB 사이에 삽입한다. 전환 전에 충분한 안정화 기간을 두고 PgBouncer 경유 운영을 검증한다.

```
변경 전: App → PostgreSQL (직접 연결)
변경 후: App → PgBouncer → PostgreSQL
```

주요 설정 항목:

| 설정 | 값 | 근거 |
|------|-----|------|
| `pool_mode` | `transaction` | session 모드는 PAUSE 중 커넥션이 해제되지 않음 |
| `default_pool_size` | HikariCP max-pool-size와 정합 | OPS-003 결과 기반으로 결정 |
| `query_wait_timeout` | HikariCP connectionTimeout 이하 | PAUSE 중 타임아웃 방지 |

#### Old DB WAL 설정

Logical Replication에는 `wal_level = logical`이 필요하다. 변경 시 PostgreSQL 재시작 필요.

```sql
-- 현재 값 확인
SHOW wal_level;

-- postgresql.conf 수정 후 재시작
-- wal_level = logical
```

#### New DB 스키마 동기화

Logical Replication은 데이터만 복제하며 DDL(스키마)은 복제하지 않는다. New DB에 스키마를 먼저 반영한다.

```bash
# Old DB에서 스키마만 덤프
pg_dump --schema-only -d moyeobab > schema.sql

# New DB에 적용
psql -d moyeobab -f schema.sql
```

### 3.3. Initial Sync (초기 데이터 복제)

Subscription 생성 시 `copy_data = true` 옵션을 사용하면 PostgreSQL이 자동으로 초기 스냅샷을 전송하고, 이후 변경사항을 이어서 복제한다.

```sql
-- Old DB: Publication 생성
CREATE PUBLICATION migration_pub FOR ALL TABLES;

-- New DB: Subscription 생성 (초기 데이터 자동 복제)
CREATE SUBSCRIPTION migration_sub
  CONNECTION 'host=<old_db_host> dbname=moyeobab user=replication_user'
  PUBLICATION migration_pub
  WITH (copy_data = true);
```

`copy_data = true`는 Old DB에 부하를 줄 수 있다. 데이터 규모가 클 경우 트래픽이 낮은 시간대에 수행하거나, `pg_dump --snapshot`과 Replication Slot을 조합하는 방법을 고려한다.

### 3.4. 시퀀스 동기화

Logical Replication은 시퀀스를 복제하지 않는다. 전환 시점에 PgBouncer PAUSE 상태에서 동기화한다.

**기본 전략: PAUSE 중 정확한 값 동기화**

```sql
-- Old DB: 모든 시퀀스 현재값 추출
SELECT sequencename, last_value FROM pg_sequences WHERE schemaname = 'public';

-- New DB: 추출한 값으로 일괄 설정
SELECT setval('meetings_id_seq', <old_last_value>);
SELECT setval('users_id_seq', <old_last_value>);
-- ... 모든 시퀀스
```

이 스크립트를 사전에 자동화해두고, 리허설에서 실행 시간을 측정한다. PAUSE 시간을 최소화하기 위함.

**Fallback: Sequence Offset**

시퀀스 동기화 스크립트 실행에 실패할 경우를 대비하여, 사전에 New DB 시퀀스를 +1,000,000 offset으로 설정해둔다. PK 타입이 BIGINT이므로 1,000,000 offset은 상한에 영향 없음.

```sql
-- New DB: Subscription 생성 전에 사전 설정 (보험)
SELECT setval('meetings_id_seq', 1000000, false);
SELECT setval('users_id_seq', 1000000, false);
```

이 offset은 정상 흐름에서는 정확한 값으로 덮어쓰기되므로 ID 갭이 발생하지 않는다. 동기화 스크립트가 실패했을 때만 offset이 적용되어 PK 충돌을 방지한다.

### 3.5. 전환 절차

Forward Replication의 Lag이 0에 수렴한 것을 확인한 후 전환을 시작한다.

```
1. PgBouncer PAUSE
   → 진행 중인 쿼리 완료 대기, 새 쿼리는 큐에 적재

2. Forward Replication Lag = 0 최종 확인
   SELECT * FROM pg_stat_replication;  -- Old DB에서

3. Forward Subscription 비활성화 (Old → New 중지)
   ALTER SUBSCRIPTION migration_sub DISABLE;  -- New DB에서

4. 시퀀스 동기화 스크립트 실행
   -- Old DB에서 현재값 추출 → New DB에 setval

5. PgBouncer backend를 New DB로 변경
   -- pgbouncer.ini의 host 변경 후 RELOAD
   -- 또는 PgBouncer 관리 콘솔에서 변경

6. Reverse Subscription 활성화 (New → Old 시작, 롤백 대비)
   ALTER SUBSCRIPTION reverse_sub ENABLE;  -- Old DB에서

7. PgBouncer RESUME
   → 큐에 쌓인 쿼리가 New DB로 전송
```

**예상 PAUSE 시간:**

| 작업 | 예상 시간 |
|------|----------|
| Lag = 0 확인 | < 1초 |
| Subscription DISABLE | < 1초 |
| 시퀀스 동기화 | 리허설 측정 필요 (테이블 수 의존) |
| PgBouncer backend 변경 | < 1초 |
| Subscription ENABLE | < 1초 |
| **합계 (목표)** | **< 10초** |

HikariCP `connectionTimeout` 이내에 완료되어야 한다 (기본 30초). 리허설에서 반드시 실측.

### 3.6. 무중단 롤백 절차

전환 후 Reverse Replication(New → Old)이 동작하고 있으므로, Old DB에는 전환 후 New DB에 쓰인 데이터가 실시간으로 동기화된다.

```
1. PgBouncer PAUSE

2. Reverse Replication Lag = 0 확인

3. Reverse Subscription 비활성화 (New → Old 중지)
   ALTER SUBSCRIPTION reverse_sub DISABLE;  -- Old DB에서

4. 시퀀스 동기화 (New → Old 방향)
   -- New DB에서 현재값 추출 → Old DB에 setval

5. PgBouncer backend를 Old DB로 변경

6. Forward Subscription 재활성화 (Old → New, 필요 시)
   ALTER SUBSCRIPTION migration_sub ENABLE;  -- New DB에서

7. PgBouncer RESUME
   → 무중단 롤백 완료, 데이터 유실 없음
```

### 3.7. 양방향 Replication 준비

전환 전에 양방향 Subscription을 미리 생성해둔다.

```sql
-- Old DB: Forward Publication (이미 생성됨)
-- CREATE PUBLICATION migration_pub FOR ALL TABLES;

-- New DB: Reverse Publication
CREATE PUBLICATION reverse_pub FOR ALL TABLES;

-- Old DB: Reverse Subscription (비활성 상태로 생성)
CREATE SUBSCRIPTION reverse_sub
  CONNECTION 'host=<new_db_host> dbname=moyeobab user=replication_user'
  PUBLICATION reverse_pub
  WITH (enabled = false, copy_data = false);
```

**Replication Loop 방지:**

양방향 Replication에서 "Old → New → Old → ..." 무한 루프가 발생할 수 있다. 하지만 PgBouncer PAUSE 중에 한쪽을 DISABLE하고 다른 쪽을 ENABLE하므로, **동시에 양방향이 활성화되는 순간이 없다**. 따라서 루프는 발생하지 않는다.

PostgreSQL 16+에서는 `origin = none` 옵션으로 복제 수신 데이터의 재전파를 방지하는 내장 기능이 있다. 현재 PostgreSQL 버전에 따라 추가 안전장치로 활용 가능.

## 4) Redis 마이그레이션 상세

### 4.1. 저장 데이터 특성

| 데이터 | TTL | 유실 영향 |
|--------|-----|----------|
| 리프레시 토큰 (로그인 세션) | 있음 | 해당 사용자 재로그인 필요 |
| 회원가입 동의 임시 데이터 | 짧음 | 사용자 재시도 |

세션 보존이 목표이므로 Redis 전환 중 토큰 유실을 최소화해야 한다.

### 4.2. 전환 방식: REPLICAOF + 승격

```
[준비]
  App → Old Redis (Master)
  New Redis = REPLICAOF Old Redis (Replica, 실시간 동기화)

[전환]
  1. Replication 완료 확인 (Lag = 0)
  2. REPLICAOF NO ONE (New Redis 독립 Master로 승격)
  3. App Redis 연결을 New Redis로 변경

[전환 후]
  App → New Redis (Master)
  Old Redis: 대기 (롤백 대비)
```

### 4.3. 전환 갭과 영향

Redis에는 PgBouncer 같은 PAUSE 메커니즘이 없다. 앱이 Redis에 직접 연결하는 구조이므로 전환 과정에서 불가피한 갭이 발생한다.

```
t=0: REPLICAOF NO ONE 실행 (New Redis 독립)
     → 이 시점부터 Old Redis의 변경은 New Redis에 반영되지 않음
t=1~3초: App Redis 연결 전환 (환경변수 변경 + 재시작 또는 동적 갱신)
     → 이 사이에 Old Redis에 쓰인 리프레시 토큰은 New Redis에 없음
```

**영향 범위**: t=0 ~ t=전환 완료(1~3초) 사이에 리프레시 토큰을 **새로 발급받은** 사용자만 영향. 기존에 발급된 토큰은 이미 양쪽에 존재하므로 문제 없음.

**완화 방안:**

| 방안 | 설명 | 영향 |
|------|------|------|
| 갭 최소화 | REPLICAOF NO ONE 직후 즉시 앱 전환 (자동화 스크립트) | 1~3초 갭 |
| 읽기 Fallback | 전환 후 New Redis에 없으면 Old Redis에서 한 번 더 조회 | 코드 수정 필요, 갭 없음 |
| 짧은 차단 | 전환 직전에 리프레시 토큰 재발급 API만 잠시 503 반환 | 사용자 수 초 후 재시도 시 성공 |

### 4.4. 롤백

Redis 데이터 특성상(세션, 짧은 TTL) PostgreSQL과 같은 무중단 롤백 구조는 불필요하다.

롤백 시나리오:
1. New Redis에 문제 발생
2. App Redis 연결을 Old Redis로 변경
3. New Redis 전환 후 발급된 리프레시 토큰은 Old Redis에 없음 → 해당 사용자 재로그인

**롤백 영향**: 전환 후 ~ 롤백 시점 사이에 발급된 리프레시 토큰 보유 사용자만 재로그인. 서비스 중단은 없음.

## 5) 사전 조사 체크리스트

마이그레이션 착수 전에 반드시 확인해야 하는 항목.

### 5.1. PostgreSQL

| 확인 항목 | 근거 | 확인 방법 |
|-----------|------|----------|
| PostgreSQL 버전 | Logical Replication은 10+ 필요 | `SELECT version()` |
| `wal_level` 현재값 | `logical`이어야 함, 변경 시 재시작 | `SHOW wal_level` |
| PK 타입 (int4/int8) | Sequence Offset 범위 결정 | Entity 클래스 또는 `\d+ 테이블명` |
| 모든 시퀀스 목록 | 동기화 스크립트 작성 | `SELECT * FROM pg_sequences` |
| 테이블 수 / 데이터 크기 | Initial Sync 소요시간 추정 | `SELECT pg_database_size('moyeobab')` |

## 6) 운영 중 주의사항

### 6.1. DDL Freeze (스키마 변경 동결)

Logical Replication은 DDL을 복제하지 않는다. 마이그레이션 기간 동안 스키마 변경(ALTER TABLE, CREATE INDEX 등)이 발생하면 양쪽 DB 간 불일치로 Replication이 중단된다.

**정책**: 마이그레이션 시작 ~ 완료(Old DB 해제)까지 **스키마 변경 금지**. 불가피한 경우 양쪽 DB에 수동으로 동일 DDL 적용.

### 6.2. WAL 디스크 관리

Logical Replication은 Replication Slot을 사용한다. New DB가 변경을 수신하지 못하면 Old DB에 WAL이 무한히 쌓인다.

```
위험 시나리오:
  New DB 장애 → Replication 중단 → Old DB WAL 폭증 → Old DB 디스크 full
```

**대비책:**
- PostgreSQL 13+: `max_slot_wal_keep_size` 설정으로 WAL 상한 지정
- Replication Lag 모니터링 알림 구성 (Grafana)
- Slot 비활성 상태 감지 시 수동/자동 제거 절차

### 6.3. PgBouncer PAUSE 중 타임아웃

PAUSE 동안 앱에서 보낸 쿼리는 PgBouncer 큐에 적재된다. PAUSE가 HikariCP `connectionTimeout`을 초과하면 앱에서 커넥션 획득 실패 에러가 발생한다.

**대비책**: 전환 스크립트를 자동화하고, 리허설에서 PAUSE → RESUME 전체 소요시간을 측정하여 `connectionTimeout` 대비 충분한 여유가 있는지 확인한다.

## 7) 리스크 매트릭스

| 리스크 | 영향도 | 대응 |
|--------|--------|------|
| Replication Lag이 0에 수렴하지 않음 | High | 전환 중단, 원인 분석 후 재시도 |
| PAUSE 중 시퀀스 동기화 실패 | High | Fallback: 사전 설정한 +1,000,000 offset 적용 |
| PgBouncer 설정 오류로 연결 장애 | High | 사전 Staging 환경에서 PAUSE/RESUME 리허설 완료 |
| WAL 디스크 폭증 | High | `max_slot_wal_keep_size` 설정 + Lag 모니터링 |
| Initial Sync 중 Old DB 부하 | Medium | 트래픽 저점 시간대에 수행 |
| DDL 변경으로 Replication 중단 | Medium | 마이그레이션 기간 스키마 변경 동결 |
| Redis 전환 갭 중 토큰 발급 | Low | 1~3초 갭, 영향 사용자 극소수 |
| PostgreSQL 버전이 Logical Replication 미지원 | Low | 사전 조사에서 확인 (10+ 필요) |
