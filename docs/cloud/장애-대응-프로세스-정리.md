## 1. 개요

본 문서는 서비스 운영 중 장애 발생 시 대응 프로세스를 정의합니다.

---

## 2. 장애 대응 프로세스

### Step 1: 장애 인지

- Discord 알림 수신 (UptimeRobot, Cron, CD 파이프라인)
- 사용자 신고
- 직접 발견

### Step 2: 장애 유형 파악

- 어떤 서비스에서 발생했는가? (moyeobab-api, recommend, DB, 외부 서비스)
- 어떤 증상인가? (헬스체크 실패, 500 에러, 연결 실패)
- 최근 배포가 있었는가?

### Step 3: 원인 판단 및 조치

| 원인 | 조치 |
|-----|------|
| 배포로 인한 장애 | 롤백 실행 |
| DB/Redis 문제 | 해당 서비스 상태 확인 및 재시작 |
| 외부 서비스 장애 (RunPod 등) | 외부 상태 확인, Fallback 동작 확인 |
| 원인 불명 | 서버 상태 진단 스크립트 실행 |

### Step 4: 롤백 실행 (배포 원인인 경우)

```bash
# moyeobab-api
/home/ubuntu/scripts/rollback-moyeobab-api.sh

# recommend
/home/ubuntu/scripts/rollback-recommend.sh
```

### Step 5: 복구 확인

- 헬스체크 정상 여부
- 핵심 API 수동 테스트 (투표, 결과 조회)

### Step 6: 공유 및 기록

- Discord에 장애 상황 및 복구 완료 공유
- 장애 원인 분석 및 기록
