# 모임 참여 집중 부하 테스트 결과보고서

## 1. 테스트 개요

**상황**: 호스트가 카톡에 초대 코드를 뿌리면, 그룹원 5~10명이 거의 동시에 참여 버튼을 누름

본 테스트는 '모임 참여'와 '모임 상세 조회'라는 서비스의 핵심 기능을 대상으로, 사용자 유입이 급증하는 상황에서 시스템이 얼마나 안정적으로 버틸 수 있는지 검증하기 위해 수행되었습니다.

### 테스트 대상 API

| Method | Endpoint | 설명 | 비중 |
|--------|----------|------|------|
| POST | /api/v1/participate_meetings | 초대 코드로 모임 참여 | 40% |
| GET | /api/v1/meetings/{meetingId} | 참여 후 모임 상세 조회 | 60% |

### 왜 병목이 예상되는가:

같은 모임에 동시 참여 시 참여자 수 증가 로직에서 DB 락 경합이 발생할 수 있다. 특히 점심시간에 수십 개 모임에서 이게 동시에 일어나면 커넥션 풀이 부족해질 수 있다.


### 테스트 환경:

| 항목 | 내용 |
|------|------|
| 테스트 도구 | k6 |
| 테스트 시간 | 12분간 지속 부하 |
| 테스트 횟수 | 2회 (3,000 VU / 2,500 VU) |

## 4. 심층 병목 분석

### 4.1 네트워크 계층 (Blocked & TLS Handshake)

전체 지연 시간 중 **네트워크 연결 설정**에 드는 비용이 압도적입니다.

| 테스트 | blocked P95 | TLS P95 | 합계 |
| --- | --- | --- | --- |
| 1차 (3,000 VU) | 4s | 3s | 7s |
| 2차 (2,500 VU) | 1s | 1s | 2s |

### 원인: Nginx와 클라이언트 간에 **Keep-alive가 제대로 동작하지 않아**, 모든 요청마다 "새로운 통로"를 뚫고 "보안 인증(TLS)"을 하느라 시간을 소비하고 있습니다.

### 증거: 일부 요청이 최대 59초까지 대기(Max Blocked)한 점은 커넥션 풀이 완전히 고갈되었음을 의미합니다.

---

### 4.2 서버 애플리케이션 지연 (Waiting)

| 테스트 | avg | P95 | max |
| --- | --- | --- | --- |
| 1차 (3,000 VU) | 551ms | 1s | 1m |
| 2차 (2,500 VU) | 190ms | 683ms | 30s |

### 원인: DB 커넥션 풀(HikariCP) 및 Tomcat 스레드가 한계치에 도달하여, 내부 로직 처리를 위한 자원을 할당받지 못하고 대기열에서 기다리는 상태입니다.

### 증거: VU가 2,500에서 3,000으로 늘어날 때, 순수 서버 처리 시간이 683ms → 1s로 **급증**했습니다.

## 5. 시스템 임계점 판정

| VU | P95 응답시간 | 에러율 | 판정 |
| --- | --- | --- | --- |
| 2,500 | 768ms | 0.1% | 안정 |
| 3,000 | 1s | 6.7% | 한계 초과 |

### 현재 시스템 한계치: 2,500 VU 부근

2,500 → 3,000 VU 증가 시:

- 에러율 **67배 증가** (0.1% → 6.7%)
- P95 응답시간 **30% 증가** (768ms → 1s)
- 서버 처리시간 **47% 증가** (683ms → 1s)

# 투표 제출 동시 집중 테스트 결과보고서

## 1. 테스트 개요

### 상황: 투표 마감 5분 전, 아직 투표 안 한 사람들이 "아 맞다 투표" 하면서 동시에 제출

본 테스트는 '투표 후보 조회', '투표 제출', '투표 현황 폴링'이라는 투표 핵심 기능을 대상으로, 투표 마감 직전 동시 접속이 폭주하는 상황에서 시스템이 얼마나 안정적으로 버틸 수 있는지 검증하기 위해 수행되었습니다.

### 테스트 대상 API:

| Method | Endpoint | 설명 | 비중 |
|--------|----------|------|------|
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/candidates | 투표 후보 목록 조회 | 30% |
| POST | /api/v1/meetings/{meetingId}/votes/{voteId}/submissions | 투표 제출 | 40% |
| GET | /api/v1/meetings/{meetingId}/votes/{voteId}/status | 투표 현황 폴링 | 30% |

### 왜 병목이 예상되는가:

투표 제출은 쓰기 작업이라 읽기보다 무겁다. 같은 투표에 여러 명이 동시 제출하면 중복 체크 + 집계 로직에서 동시성 이슈가 생길 수 있다. 그리고 status 폴링이 계속 들어오면서 읽기 부하까지 겹친다.

### 테스트 환경:

| 항목 | 내용 |
|------|------|
| 테스트 도구 | k6 |
| 테스트 시간 | 10분간 지속 부하, 250개 모임 |
| 테스트 횟수 | 2회 (2,000 VU) |

## 4. 개선 방안

| **개선 항목** | **대상 병목** | **핵심 포인트** |
| --- | --- | --- |
| **Nginx keepalive** | **blocked** | 새로운 연결을 맺는 '비용' 자체를 줄이는 가장 기본적이면서 효과적인 단계. 
연결을 재사용하여 대기 시간을 즉각적으로 줄여줌. |
| **SSL Session Cache** | **TLS** | 암호화 통신(HTTPS)에서 가장 무거운 과정인 'Handshake' 결과를 저장함. 
재방문 시 복잡한 계산 과정을 생략하므로 CPU 부하와 지연 시간을 동시에 잡음. |
| **HTTP/2 활성화** | **blocked + TLS** | 한 번 맺은 연결에서 여러 자원을 동시에 주고받는 기술. 
위에서 언급한 다져진 연결의 효율을 증가시킴. |

---

## 5. 결론

주요 병목 구간: 네트워크 레이어 Connection Pool 지연 및 TLS Handshake

이번 부하 테스트 결과, 서버 자체의 처리 능력(Waiting Time)은 안정적인 수준을 유지하고 있습니다. 
하지만 트래픽이 몰리는 시점에서 클라이언트 대기 시간(Blocked)이 길어지는 현상이 확인되었습니다.

### 개선할 부분

- **Nginx Keepalive 설정 최적화:** 커넥션 재사용률을 높여 클라이언트의 `blocked` 시간을 즉각적으로 단축합니다.
- **SSL Session Cache 활성화:** 반복되는 TLS 핸드셰이크 부하를 줄여 암호화 통신 효율을 극대화합니다.

### 기대 효과
위 두 가지 설정 최적화만으로도 네트워크 오버헤드가 크게 줄어들어, **P95 지연 시간을 현재보다 낮은 수준으로 안정화**할 수 있을 것으로 판단됩니다.
