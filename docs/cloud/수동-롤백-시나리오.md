## 1. 개요

자동 롤백이 실행되지 않았거나 실패한 경우, 수동으로 롤백을 수행하는 절차를 정의합니다.

---

## 2. 수동 롤백 트리거 조건

- 자동 롤백 실패 알림 수신
- CD 완료 후 (Smoke Test 통과) 운영 중 장애 발생 → 배포가 원인으로 판단될 때

## 4. 롤백 실행 방법

### 4.1 선택한 방법: 서버에서 직접 스크립트 실행

SSH로 EC2에 접속하여 롤백 스크립트를 직접 실행하는 방식입니다.

| 항목 | 내용 |
|-----|------|
| 적합한 규모 | 작은 규모, 단순한 구조 (MAU 500 수준) |
| 장점 | 빠름, 단순함, 추가 설정 불필요 |
| 단점 | SSH 접속 권한 필요 |

### 4.2 대안: workflow_dispatch (참고)

GitHub Actions에서 수동 트리거 버튼으로 롤백 워크플로우를 실행하는 방식입니다.

| 항목 | 내용 |
|-----|------|
| 적합한 규모 | 팀 규모가 크고 서버 접속 권한 분리가 필요할 때 |
| 장점 | SSH 접속 권한 없이 롤백 가능, 실행 이력 GitHub에 기록 |
| 단점 | 직접 SSH보다 느림 (30초~1분), GitHub 장애 시 사용 불가 |
| 비고 | CI/CD 파이프라인과 별개로 실행됨 |

※ 현재는 서버 직접 스크립트 실행 방식을 사용하며, 팀 규모 확장 시 workflow_dispatch 도입 고려

## 5. 서버 구성 현황

### 5.1 EC2 내부 서비스

| 서비스 | 설명 | 롤백 대상 |
|--------|-----|----------|
| moyeobab-api.service | Spring Boot API 서버 | O |
| recommend.service | RunPod 연동 클라이언트 | O (별도 배포) |
| nginx.service | 리버스 프록시 | X |
| postgresql@16-main.service | PostgreSQL DB | X |
| redis-server.service | Redis (로그인 state 검증) | X |

※ 코드 배포가 없는 인프라 서비스(DB, 캐시, 프록시)는 롤백 대상에서 제외

### 5.2 외부 서비스 (롤백 대상 아님)

| 서비스 | 설명 | 장애 시 대응 |
|--------|-----|-------------|
| RunPod | GPU 기반 추천 엔진 (외부) | 별도 장애 대응 절차 |
| 음식점 API(외부) | 음식점 데이터 제공 | Fallback 또는 대기 |

## 7. 수동 롤백 절차

### 7.1 moyeobab-api 롤백 (스크립트 사용)

1. SSH로 EC2 서버에 접속
2. 롤백 스크립트 실행
   ```bash
   cd /home/ubuntu/scripts
   ./rollback-moyeobab-api.sh
   ```
3. 확인 프롬프트에 `y` 입력
4. 롤백 완료 메시지 확인

### 7.2 moyeobab-api 롤백 (수동 명령어)

스크립트 실행이 불가능한 경우 아래 명령어를 순서대로 실행합니다.

1. SSH로 EC2 서버에 접속
2. 현재 서비스 중지
   ```bash
   sudo systemctl stop moyeobab-api
   ```
3. 백업 JAR로 교체
   ```bash
   cp /home/ubuntu/app/backup/app.jar.backup /home/ubuntu/app/app.jar
   ```
4. 서비스 재시작
   ```bash
   sudo systemctl restart moyeobab-api
   ```
5. 서비스 상태 확인
   ```bash
   systemctl status moyeobab-api
   ```
6. 헬스체크 확인
   ```bash
   curl https://api.moyeobab.com/actuator/health
   ```
### 7.3 recommend 서비스 롤백 (스크립트 사용)

1. SSH로 EC2 서버에 접속
2. 롤백 스크립트 실행
   ```bash
   cd /home/ubuntu/scripts
   ./rollback-recommend.sh
   ```
3. 확인 프롬프트에 `y` 입력
4. 롤백 완료 메시지 확인

### 7.4 recommend 서비스 롤백 (수동 명령어)

스크립트 실행이 불가능한 경우 아래 명령어를 순서대로 실행합니다.

1. SSH로 EC2 서버에 접속
2. 서비스 중지
   ```bash
   sudo systemctl stop recommend
   ```
3. 이전 커밋으로 롤백
   ```bash
   cd /home/ubuntu/13-team-project-ai/services/recommend
   git log --oneline -5  # 롤백할 커밋 확인
   git reset --hard HEAD~1
   ```
4. 서비스 재시작
   ```bash
   sudo systemctl restart recommend
   ```
5. 서비스 상태 확인
   ```bash
   systemctl status recommend
   ```
6. 헬스체크 확인
   ```bash
   curl http://127.0.0.1:8000/health
   ```

### 7.5 의존 서비스 확인

롤백 후에도 문제가 지속되면 의존 서비스 상태를 확인합니다.

```bash
# 전체 서비스 상태 확인
systemctl status nginx postgresql@16-main redis-server recommend

# Redis 연결 확인
redis-cli ping

# PostgreSQL 연결 확인
sudo -u postgres psql -c "SELECT 1;"
```

### 7.6 원격 롤백 실패 시 조치

1. 서버에 직접 접속하여 상태 확인
   ```bash
   # 디스크 확인
   df -h
   
   # 메모리 확인
   free -m
   
   # 프로세스 확인
   ps aux | grep java
   ```
2. 문제 원인 파악 및 해결
3. 해결 후 7.1 또는 7.2 절차 재수행
4. 해결 불가 시 AWS 콘솔에서 인스턴스 재부팅

## 10. 비고

- 백업 JAR는 CD 파이프라인에서 배포 시 자동으로 생성됩니다 (직전 버전 1개 보관).
- moyeobab-api와 recommend는 별도 배포되므로, 장애 원인에 따라 롤백 대상을 구분해야 합니다.
- RunPod 장애 시 recommend 서비스 롤백은 무의미하며, RunPod 상태 복구를 기다리거나 Fallback 처리해야 합니다.
- Nginx, PostgreSQL, Redis는 설정 변경이 있었을 경우에만 별도 조치합니다.
