# 인스턴스 사양 선정

> 작업자 메모. 추후 본문 정리 시 참고용 초안.

## 결론

**EC2 t3.small** (ARM t4g.small도 후보였을 가능성)

## 선정 근거

### 트래픽 패턴

- 피크가 강한 패턴 가정 (점심·저녁 시간대 집중)
- 그 외 시간은 거의 유휴 상태로 추정
- 사용자당 평균 10 req/day 가정
- MVP 단계 100명 수준 가정 (부트캠프 인원 ~100명 기반)
- → 버스터블(T 시리즈)이 적합하다는 판단

### 인스턴스 후보

micro / small / medium / large 검토. 결과적으로 small 선택.

- small 선택 이유: 좀 부족한 감은 있음. 부하테스트 자료에서 근거 확인 필요.
- swap 사용 불가: Redis가 들어가서 swap 사용 불가능한 상황 (개발 측 요구사항)

## 한계 / 향후 보완

- **부하테스트의 한계**: 설계 단계에서 한 부하테스트는 개인 프로젝트 단순 REST API 대상이라 실제 서비스와 차이가 있을 가능성. 의미가 제한적.
- **개발 중 실측 필요**: 빠른 모니터링 환경 구축 후 실측해야 함.

## 메모

- ARM(t4g.small)을 선택했을 가능성 — 실제 사양 재확인 필요.
- small이 micro 대비 왜 적정선이었는지 부하테스트 결과로 보강 필요.
