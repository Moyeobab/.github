

## 1. 테스트 목적

수동 롤백 시나리오 정의서와 롤백 스크립트가 실제 운영 환경에서 정상 동작하는지 검증

---

## 2. 테스트 환경

| 항목 | 내용 |
|-----|------|
| 인스턴스 | EC2 |
| 서비스 | moyeobab-api.service |
| 배포 경로 | /home/ubuntu/app/app.jar |
| 백업 경로 | /home/ubuntu/app/backup/app.jar.backup |
| 스크립트 경로 | /home/ubuntu/scripts/ |

### 2.1 선택한 방법: 서버에서 직접 스크립트 실행

SSH로 EC2에 접속하여 롤백 스크립트를 직접 실행하는 방식입니다.

| 항목 | 내용 |
|-----|------|
| 적합한 규모 | 작은 규모, 단순한 구조 (MAU 500 수준) |
| 장점 | 빠름, 단순함, 추가 설정 불필요 |
| 단점 | SSH 접속 권한 필요 |

### 2.2 대안: workflow_dispatch (참고)

GitHub Actions에서 수동 트리거 버튼으로 롤백 워크플로우를 실행하는 방식입니다.

| 항목 | 내용 |
|-----|------|
| 적합한 규모 | 팀 규모가 크고 서버 접속 권한 분리가 필요할 때 |
| 장점 | SSH 접속 권한 없이 롤백 가능, 실행 이력 GitHub에 기록 |
| 단점 | 직접 SSH보다 느림 (30초~1분), GitHub 장애 시 사용 불가 |
| 비고 | CI/CD 파이프라인과 별개로 실행됨 |

※ 현재는 서버 직접 스크립트 실행 방식을 사용하며, 팀 규모 확장 시 workflow_dispatch 도입 검토

## 4. 테스트 시나리오 및 결과

### 4.1 시나리오 A: 정상 롤백 테스트

**목적:** 롤백 스크립트가 정상적으로 동작하는지 확인

**수행 절차:**
1. SSH로 EC2 서버 접속
2. `./rollback-moyeobab-api.sh` 실행
3. 확인 프롬프트에 `y` 입력
4. 롤백 완료 메시지 확인

**실행 결과:**

<img width="636" height="716" alt="moyeobab 수동 롤백 실행 결과" src="https://github.com/user-attachments/assets/618cf772-3502-4556-9276-1a96d6cc8a52" />


**측정 항목:**

| 항목 | 측정값 |
|-----|--------|
| 롤백 총 소요 시간 | 약 18초 |
| 헬스체크 통과까지 시간 | 약 15초 (6번째 시도) |

**결과:** 성공

**비고:** 헬스체크 초기 실패(502)는 Spring Boot 애플리케이션 시작 시간 때문이며 정상 동작임

---

### 4.2 시나리오 B-1: 백업 파일 없음 테스트

**목적:** 백업 파일이 없을 때 스크립트가 적절히 에러를 감지하는지 확인

**수행 절차:**
1. 백업 파일 임시 이동
   ```bash
   mv /home/ubuntu/app/backup/app.jar.backup /home/ubuntu/app/backup/app.jar.backup.temp
   ```
2. 롤백 스크립트 실행
3. 에러 메시지 확인
4. 백업 파일 원복
