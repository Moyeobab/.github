## 1. 문제 상황
현상: develop 브랜치로의 PR 생성 및 업데이트 시, GitHub Actions의 CI 파이프라인 중 CheckStyle 단계에서 빌드 실패 발생.

영향: 모든 개발팀의 코드 머지가 차단되었으며, 배포 파이프라인이 중단되어 서비스 업데이트가 불가능한 Critical 상태임.

## 2. 원인 분석
기술적 원인: GlobalExceptionHandler 클래스 내 로깅 객체(log)의 명칭이 프로젝트 정적 분석 규칙인 ConstantName 패턴과 충돌함.

패턴 요구사항: ^[A-Z][A-Z0-9](_[A-Z0-9]+)$ (모든 상수는 대문자와 언더바만 허용)

위반 사례: 소문자 log 사용.

환경적 원인: * Lombok의 @Slf4j가 생성하는 log 필드는 관습적으로 소문자를 사용하나, CheckStyle 설정이 이를 예외 처리하지 않음.

Root Module 설정 오류: CI 환경에서 참조하는 checkstyle.xml 설정이 최신 컨벤션을 반영하지 못하거나 경로 설정이 어긋나 기본 규칙이 적용됨.

## 3. 해결책
CheckStyle 규칙 완화 및 예외 등록: checkstyle.xml의 ConstantName 모듈 설정을 수정하여, 로깅 관례로 사용되는 log 명칭을 허용하도록 정규표현식을 업데이트함.

설정 파일 동기화: CI 서버뿐만 아니라 개발자 로컬 IDE(IntelliJ 등)에서도 동일한 checkstyle.xml을 강제 참조하도록 설정 가이드 배포.

## 4. 대안 검토 및 한계점
대안 1: 변수명을 LOG로 변경

장점: 규칙 수정 없이 즉각적인 빌드 통과 가능.

한계: @Slf4j 사용 시 필드명을 변경하려면 추가 어노테이션 속성이 필요하며, 프로젝트 전반의 기존 코딩 관습과 어긋나 일관성이 깨짐.

대안 2: @SuppressWarnings("checkstyle:ConstantName") 사용

장점: 특정 코드 라인에서만 규칙 무시 가능.

한계: 근본적인 설정 오류(Unable to create Root Module 등)를 해결하지 못하며, 향후 유사한 로깅 코드가 추가될 때마다 중복 작업이 발생함.

## 5. 기대 효과
CI 안정성 확보: 불필요한 스타일 규제로 인한 빌드 중단 현상을 제거하여 개발 리듬을 유지함.

유연한 컨벤션 적용: 프레임워크(Lombok, Spring) 특성과 정적 분석 도구 간의 충돌을 조정하여 실무에 적합한 코드 품질 기준 확립.

자동화 신뢰도 상승: 정적 분석이 '방해 요소'가 아닌 '품질 보증 수단'으로 인식되도록 인프라 신뢰성 개선.
