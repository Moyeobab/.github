## 1. 문제 상황
현재 프로젝트의 CI(Continuous Integration) 과정에서 정적 분석 도구인 Spotbugs가 실제 결함이 아닌 코드를 취약점으로 탐지하여 빌드가 빈번하게 중단되는 문제가 발생하고 있습니다.

구체적 사례: 스프링 프레임워크의 핵심 메커니즘인 의존성 주입(Dependency Injection) 과정에서 발생하는 필드 주입이나 생성자 구조를 '외부 영향에 취약한 상태'로 오인하는 경우입니다.

영향: 불필요한 빌드 실패로 인해 개발자의 생산성이 저하되고, 실제 중요한 보안 결함을 확인하는 과정에서 피로도가 쌓여 분석 결과에 대한 신뢰도가 낮아지고 있습니다.

## 2. 원인 분석
Spotbugs는 범용적인 Java 보안 규칙을 적용하기 때문에, 스프링과 같은 프레임워크 특유의 관습을 '취약점'으로 판단할 때가 있습니다. 예를 들어, 가변적인 필드나 외부 노출 가능성이 있는 객체 구조를 엄격하게 검사하면서 발생하는 오탐(False Positive)입니다.

## 3. 해결책: exclude.xml을 통한 필터링 설정
코드 수준의 대응보다는 프로젝트 전역 설정을 통해 불필요한 검사 과정을 구조적으로 제거하기로 결정했습니다.

적용 방법: spotbugs-exclude.xml 파일을 생성하여 특정 버그 패턴(Bug Pattern)이나 특정 패키지/클래스를 검사 대상에서 제외합니다.

## 4. 대안 검토 및 한계점
@SuppressFBWarnings 사용:

장점: 특정 메서드나 클래스 단위로 세밀하게 제어할 수 있습니다.

단점(선택하지 않은 이유): 비즈니스 로직에 정적 분석 도구 종속적인 어노테이션이 추가되어 코드가 지저분해집니다. 또한, 동일한 패턴이 발생할 때마다 매번 추가해야 하므로 근본적인 유지보수 비용을 해결하지 못합니다.

## 5. 기대 효과
CI 속도 향상: 불필요한 분석 단계를 건너뜀으로써 전체적인 빌드 및 배포 사이클이 단축됩니다.

분석 신뢰도 확보: '진짜 문제'가 있을 때만 빌드가 실패하므로, 팀 내에서 정적 분석 결과에 대한 경각심과 신뢰를 회복할 수 있습니다.

DevOps 최적화: 인프라 자원의 낭비를 줄이고, 엔지니어가 코드 품질의 본질에 집중할 수 있는 환경을 조성합니다.
